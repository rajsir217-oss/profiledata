Design rationale

Hypothesis: Users open the dashboard intending to do something — search, reply, approve a request. Optimize for action-completion, not stat-browsing.

  • Hero = newest match in your default saved search, not a search bar. Search has its own page (/search); the dashboard hero shows the top-1 result of the user's default saved search sorted by joinedAt desc — piggybacks on the existing saved-search digest cache, no separate curation job needed. Inline View / Favorite / Message / Skip; Skip walks to the next-newest. Fallback chain when a search has no new joiners: try next-most-recent saved search (by lastRunAt), then the next, etc. Only if all saved searches are empty does the hero show an empty state. Subtitle ("newest match in X") always reflects which search produced the current pick. The saved-search chip strip below provides 1-click navigation to /search with per-search new-match counts.
  • "What needs your attention" task cards replace banner pile-up. Each card is a job-to-be-done with a count badge.
  • Stats compressed from 9 cards to 4 actionable ones. Red dot on tiles with new activity.
  • Inline message previews — most-frequent action surfaced inline, no extra click.
  • Side rail = passive nudges (profile strength, active polls with multi-poll navigator, saved searches with "new" counts). The polls widget displays one poll at a time with ◀/▶ navigation (no accordion); voting auto-advances to the next unanswered poll. Header badge shows total active poll count.
  • 6 banners → 1 notification drawer (bell, top-right). Click to open.

Trade-offs: Less data density than current. Power users who want full activity logs may need to drill into sub-pages. No accordions on this page.
