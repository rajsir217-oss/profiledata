Dashboard redesign — 4 mockups

Self-contained HTML mockups. Click any card to open. No build, no server, no React. Realistic mock data; you can interact with the notification drawer, scrollers, etc. All four use the same theme variables as production.

📋 Why redesign — audit of current dashboard

  • 6 conditional banners can stack vertically and push real content below the fold (MFA, pause, photos, invites, reconnect, MFA enable).
  • 9 stat cards at top — most duplicate counts already shown in the accordion section pills below.
  • 3 collapsible accordions (Data Requests / My Activities / Prospects) — heavy navigation cost; most users see a wall of toggles.
  • No fresh content: zero new-match feed, zero saved-search preview, zero L3V3L suggestions surfaced.
  • No quick search: primary action of the app requires navigating to /search.
  • No timeline / "since last visit" framing — user can't tell what's new.
  • Profile completeness only surfaces when photos are missing (one-shot banner).
  • Poll widget buried inline among stat cards — easy to miss.

🛣️ Recommended next steps

  1. If risk-averse → ship Mockup D first. 1-2 days, zero new APIs, zero behavior change. Solves the "too much scrolling" pain without touching anything else. Buys time to validate A/B/C with real users in parallel.
  2. If ambitious → pick a direction (or hybrid) — A/B/C aren't mutually exclusive. Common hybrid: Mockup A's hero search + Mockup C's saved-search rows below + Mockup B's notification drawer.
  3. Validate with 2-3 users. Show the 4 mockups, ask "which most matches what you actually do here?" Their answer often surprises you vs. what we'd guess.
  4. Pick the lowest-risk slice to ship first — the notification drawer is shared across all 3 mockups and reduces banner pile-up immediately. That alone is a 1-day shippable improvement.
  5. Write the data contracts before code. If we go with B or C, define the new endpoints (e.g. GET /dashboard/timeline, GET /saved-searches/{id}/preview) so backend and frontend can move in parallel.
  6. Don't ship in place. Add a feature flag or /dashboard?v=new query param so users can switch between old + new while we iterate.
